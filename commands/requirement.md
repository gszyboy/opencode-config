description: 从用户需求生成结构化需求文档，输出五层结构：模块 → 功能项 → 具体需求 → 技术原理 → 模块关联
agent: plan
---

请帮助用户将需求转化为完整的结构化需求文档。

## 输入

用户的简单描述或一句话需求。

## 工作流程

### 第一步：需求澄清

向用户提问确认关键信息：

```
## 需求澄清

在生成完整需求文档之前，请确认以下信息：

### 1. 核心用户
- 谁会使用这个系统？
- 用户的技术水平？

### 2. 核心功能（请按优先级排序）
- P0（必须有）：？
- P1（应该有）：？
- P2（可以有）：？

### 3. 平台
- [ ] Web
- [ ] 小程序
- [ ] H5
- [ ] APP

### 4. 数据存储
- 需要数据库吗？什么类型？
- 需要用户账号体系吗？

### 5. 第三方集成
- [ ] 微信登录
- [ ] 支付宝支付
- [ ] 短信验证码
- [ ] 其他：

### 6. 特殊要求
- 性能要求？
- 安全要求？
- 兼容要求？
```

### 第二步：生成五层结构需求文档

生成 `docs/requirements.md`，包含以下结构：

```markdown
# [项目名] 需求规格说明书

## 项目概述

### 背景
[简要描述项目背景和目标]

### 核心用户
[目标用户群体]

### 平台
- [ ] Web
- [ ] 小程序
- [ ] H5
- [ ] APP

---

## 模块划分

### M1：[模块名]

**模块描述**：
[模块的功能概述]

**优先级**：P0/P1/P2

**模块责任**：
- 负责...
- 不负责...

---

## 功能详细说明

### 模块 1：M1

#### 功能 1.1：[功能名]

**用户故事**：
> 作为 [角色]，我想要 [功能]，以便 [价值]。

**功能描述**：
[详细的功能描述]

**用户操作流程**：
1. 用户点击...
2. 系统...
3. 用户...

**界面元素**：
- 输入框：[名称] - [类型] - [验证规则]
- 按钮：[名称] - [状态] - [操作]
- 列表：[字段] - [排序] - [分页]

**业务规则**：
- 规则 1
- 规则 2

**边界情况**：
- 空数据时显示...
- 加载中显示...
- 错误时显示...

**验收标准**：
- [ ] 可以 [操作]
- [ ] 显示 [结果]
- [ ] 当 [条件] 时 [行为]

---

## 技术规格

### 模块依赖关系

```
[M1] → [M2]
[M1] → [认证模块]
[M2] → [数据库]
```

### 数据模型

#### [表名]
| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 主键 |
| ... | ... | ... |

### 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端框架 | FastAPI | - |
| 数据库 | PostgreSQL | - |
| 前端框架 | Vue 3 | - |
| UI | Tailwind CSS | - |

---

## 非功能需求

### 性能
- API 响应时间：< 500ms
- 页面加载时间：< 2s

### 安全
- 用户密码加密存储
- API 需要认证

### 兼容性
- Chrome/Firefox/Safari 最新两个版本

---

## 边界定义

### 包含范围
- [ ] 功能 A
- [ ] 功能 B

### 不包含范围
- [ ] 支付功能（后续版本）
- [ ] 管理后台（单独项目）

---

## 优先级排序

### P0 - MVP 必须
1. [功能 1]
2. [功能 2]

### P1 - 重要功能
1. [功能 3]

### P2 - 增强功能
1. [功能 4]
```

### 第三步：生成开发计划

基于需求文档，调用 `@plan-decomposer` 生成 `docs/development-plan.md`

## 输出文件

- `docs/requirements.md` - 五层结构需求文档
- `docs/development-plan.md` - 开发计划

## 约束

- 保持简洁，不做过度设计
- MVP 只包含 P0 功能
- 每个模块独立可开发
- 先提问再生成，不要假设用户需求

## 不要做的事

- ❌ 不要假设用户需求，自己脑补
- ❌ 不要添加需求文档没提到的功能
- ❌ 不要使用模糊的描述（如"用户体验好"）
- ❌ 不要一次性生成完整文档，分步确认
