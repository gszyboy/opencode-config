# 增量式文档维护机制

## 核心思想

```
❌ 以前的做法：
  发现问题 → 重写整个计划文档 → 工作量大 → 不更新 → 文档过时

✅ 现在的做法：
  发现问题 → 追加到变更记录 → 工作量小 → 保持同步 → 文档始终最新
```

---

## 使用场景

### 场景 1：开发中发现需求遗漏

```
发现：需求文档没提到"用户退出后要清除 Token"

处理：
1. 在 CHANGELOG.md 中追加
2. 在开发计划中补充任务
3. 继续开发，不中断

不做什么：
❌ 不重写整个需求文档
❌ 不重新分解所有模块
```

### 场景 2：开发中发现技术方案不可行

```
发现：原本计划用 LocalStorage，但小程序不支持

处理：
1. 在 CHANGELOG.md 记录
2. 调整技术方案
3. 继续开发

不做什么：
❌ 不质疑 AI 的能力
❌ 不从头讨论技术选型
```

### 场景 3：发现任务边界不清

```
发现："用户管理模块"包含了太多功能

处理：
1. 在 CHANGELOG.md 拆分
2. 重新定义模块边界
3. 调整后续任务

不做什么：
❌ 不重新规划整个项目
```

---

## 操作流程

### 第一步：评估影响

| 影响程度 | 定义 | 处理方式 |
|----------|------|----------|
| **轻微** | 不影响核心流程 | 记录 + 继续 |
| **中等** | 需要调整任务 | 记录 + 调整 + 继续 |
| **重大** | 影响项目方向 | 讨论 + 决策 + 更新 |

### 第二步：记录变更

在 `docs/CHANGELOG.md` 中追加：

```markdown
### 2026-03-22 变更 #1

**发现**：原计划用 LocalStorage 存储 Token
**问题**：小程序不支持 LocalStorage
**调整**：改用 uni.setStorageSync
**影响**：修改 utils/auth.js
```

### 第三步：更新相关文档

只更新变化的部分：

```
涉及什么 → 更新什么
   ↓
原计划文档 → 补充遗漏的任务
技术文档 → 修正技术方案
但不重写无关部分
```

## 文档分层策略

不要把所有东西写在一个大文档里：

```
docs/
├── requirements.md      # 原始需求（基本不改）
├── development-plan.md  # 开发计划（只增不减）
├── progress.json        # 开发进度状态（JSON）
├── CHANGELOG.md        # 变更记录（核心）
└── decisions.md        # 技术决策记录
```

### 各文档职责

| 文档 | 内容 | 更新频率 |
|------|------|----------|
| `requirements.md` | 原始需求 | 几乎不改 |
| `development-plan.md` | 开发计划 | 任务完成时更新状态 |
| `progress.json` | 开发进度状态 | 任务开始/完成时更新 |
| `CHANGELOG.md` | 变更记录 | 发现问题时更新 |
| `decisions.md` | 技术决策 | 做出决策时记录 |

---

## 决策原则

### 立即处理的情况
- 影响功能正常运行的错误
- 阻塞开发的问题
- 安全漏洞

### 记录暂缓的情况
- 非核心功能的遗漏
- 可以绕过的技术问题
- 需要讨论的架构调整

### 记录但后续处理的情况
- 代码质量问题
- 重复造轮子
- 可以优化但不紧急的

## 模板

### 快速记录模板

```
### [日期] 变更 #N

**类型**：[遗漏/错误/调整]
**位置**：[哪个模块/文件]
**发现**：[具体问题]
**调整**：[如何处理]
**影响**：[影响范围]
```

### 详细记录模板

```markdown
### [日期] 变更 #N

**发现时间**：开发 [模块名] 时
**发现人**：AI/用户
**问题类型**：遗漏/错误/调整/新增

**原始计划**：
> [原来是怎么说的]

**实际情况**：
> [开发中发现的问题]

**原因分析**：
> [为什么会出现这个问题]

**调整方案**：
> - 任务调整：[具体调整]
> - 文档更新：[需要更新的文档]
> - 代码修改：[需要修改的文件]

**影响评估**：
- 时间影响：
- 范围影响：

**决策**：
- [ ] 立即修复
- [ ] 记录暂缓
- [ ] 需要讨论
```

---

## 不需要更新文档的情况

- 只是代码实现方式的微调
- 不影响功能的格式化修改
- 临时的调试代码（及时删除）

## 与 /dev 命令配合

### 发现问题时

使用 /dev 开发过程中发现问题：
1. 评估影响程度（轻微/中等/重大）
2. 在 `docs/CHANGELOG.md` 追加记录
3. 在 `docs/progress.json` 更新任务状态（如需）
4. 继续开发

### 任务状态更新

如果问题影响任务进度，更新 `docs/progress.json`：

| 情况 | 操作 |
|------|------|
| 问题导致任务无法继续 | 将任务状态改为 `blocked`，添加 notes 说明 |
| 问题需要延后处理 | 在 notes 中记录，继续其他任务 |
| 问题已解决 | 清除 notes，状态改回 `in_progress` |

### 示例：更新 progress.json 记录问题

```json
{
  "tasks": [
    {
      "id": 2,
      "title": "用户登录功能",
      "status": "blocked",
      "notes": "发现原计划用 LocalStorage，但小程序不支持，需要调整方案"
    }
  ]
}
```

---

## 保持文档同步的习惯

1. **每天结束时**：检查 progress.json 进度
2. **每次发现问题**：立即记录到 CHANGELOG
3. **每周回顾**：检查 CHANGELOG，更新开发计划状态
4. **每月整理**：归档已完成的变更，简化文档
