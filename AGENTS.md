# 全局开发规范

## 语言
- 保持对话为中文
- 你必须始终使用简体中文回复。所有思考过程、推理步骤、todo列表和输出都必须使用简体中文。

## 核心原则
1. 简单优先，不提前抽象
2. 每个改动必须有明确目的
3. 先让它跑起来，再优化
4. AI 加速开发，但不替代决策
5. 不确认就动手，比不动更危险

## 行为准则

### 1. Think Before Coding
**不假设、不隐藏困惑、不绕过问题。**

- 有疑问必须先问清楚，不能猜着做
- 有多种解释时，先呈现再选择，不能静默Pick
- 发现更简单的方案要说出来，该 Push Back 就 Push Back
- 遇到不清楚的，直接停下来问

### 2. Simplicity First
**最小代码解决问题，不写预期外的东西。**

- 不做没要求的功能
- 不为单次使用创建抽象
- 不加没要求的"灵活性"
- 不处理不可能发生的错误场景
- 200行能搞定就别写300行

### 3. Surgical Changes
**只碰必须碰的，只清理自己造成的烂摊子。**

编辑现有代码时：
- 不"顺便"改善相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格，即使你有更好的写法
- 发现无关死代码 → 提一句，不删

你的改动产生的孤儿代码：
- 移除你改动导致的无用 import/变量/函数
- 不移除改动前就存在的死代码（除非明确要求）

**检验标准：每行改动都能追溯到用户需求。**

### 4. Goal-Driven Execution
**定义成功标准，循环验证直到达成。**

任务转换示例：
- "加验证" → "先写无效输入的测试，再让测试通过"
- "修 Bug" → "先写复现测试，再让测试通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务要有简短计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

## Change Budget（强制，所有项目通用）
- 修改文件数：≤ 3
- 修改行数：≤ 80（不含格式化）
- 不引入新依赖（除非明确要求）
- 不修改任务范围之外的文件

## 禁止事项
- ❌ 不创建 service/repository 层（除非有 3+ 重复代码）
- ❌ 不添加"为未来扩展"的代码
- ❌ 不重构未涉及的文件
- ❌ 不创建抽象基类或接口（除非有必要）
- ❌ 单次任务不追求完美
- ❌ 不过度工程（YAGNI 原则）
- ❌ 不静默做选择，有歧义必须先问

## 安全规范
- ❌ 禁止在代码中硬编码密钥、密码、token
- ❌ 禁止提交 .env 文件到 Git
- 使用环境变量或 {env:VAR_NAME} 语法

### 目录安全
**危险目录禁止操作**：以下目录不适合执行代码操作，发现后必须立即警告用户并拒绝创建/修改任何文件：
- `~` 用户根目录
- `/` 根目录
- `/etc`、`/var`、`/usr`、`/boot`、`/sys`、`/proc`、`/dev` 系统目录

发现用户在危险目录时，必须：
1. 立即通过 TUI toast 或回复提醒用户
2. 拒绝创建、修改任何代码文件
3. 建议用户切换到正规项目目录

## Git 规范
- Commit 信息简洁明了
- 每次 commit 只做一件事
- 提交前检查敏感信息

## 开发节奏
1. 明确任务范围 → **不确定就问**
2. 设置 Change Budget
3. 复杂任务先 Plan Mode 分析
4. 小步快跑，不追求一次到位
5. 完成后简单验证

### 已有项目核心规则

**50行平衡模式**：
- 改动 ≤ 50行
- 文件 ≤ 3个
- 不引入新依赖

**改动红线（绝对不改，除非明确授权）**：
1. 数据库结构/ORM模型
2. 认证鉴权逻辑
3. 支付财务代码
4. 核心业务算法

**介入级别**：
- L1 只读：刚接手、不确定风险
- L2 诊断：定位问题、分析原因
- L3 小改：改一个函数/一行配置（≤10行）
- L4 增量：修复bug、加小功能（≤50行）
- L5 重构：较大改动（分阶段授权）
