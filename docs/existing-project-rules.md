# 已有项目AI协作规范

## 核心原则

**保守优先**：宁可多问，不可乱改
**50行平衡**：每次改动 ≤ 50行，文件 ≤ 3个
**红线必守**：数据库结构、认证鉴权、支付财务绝对不改

---

## 介入级别

| 级别 | 场景 | AI权限 |
|------|------|--------|
| L1 只读 | 刚接手、不确定风险 | 只读、回答问题 |
| L2 诊断 | 定位问题、分析原因 | 分析并输出报告 |
| L3 小改 | 改一个函数/一行配置 | ≤10行，逐次确认 |
| L4 增量 | 修复bug、加小功能 | ≤50行，任务完成review |
| L5 重构 | 较大改动 | 分阶段授权，明确边界 |

---

## 改动红线（绝对不改）

除非用户明确授权，否则绝对不修改：

1. **数据库结构**
   - ORM模型文件（models.py）
   - 数据库迁移文件
   - 数据库连接配置

2. **认证鉴权**
   - 登录/注册逻辑
   - 权限控制代码
   - Token/密钥处理

3. **支付财务**
   - 支付接口
   - 财务计算逻辑
   - 订单处理核心

4. **核心业务**
   - 核心算法
   - 关键业务逻辑

---

## 评估流程

### /assess-project

在项目目录下执行，AI将：

1. 结构扫描 → 技术栈识别
2. 风险评估 → 核心业务识别
3. 输出项目评估报告

### 评估输出格式

```markdown
# 项目评估报告

## 基本信息
- 项目类型：[自研/二次开发/开源改造]
- 技术栈：[列出]
- 代码规模：[行数估算]

## 风险矩阵
| 区域 | 风险 | 说明 |
|------|------|------|
| xxx/ | 高 | ... |

## 推荐级别：L4

## 红线清单
1. models/user.py
2. auth/*.py
3. ...

## 下一步建议
...
```

## 开发流程

### /incremental-dev [任务]

已有项目的增量开发命令：

**改动前**：
1. 确认改动范围
2. 检查红线
3. 自动git backup

**改动中**：
- 每次≤50行
- 涉及多文件先说明
- 遇红线立即停

**改动后**：
- 说明改动内容
- 提供验证方法
- 确认功能正常

---

## 快速回滚

```bash
# 自动备份
git tag "backup/before-$(date +%Y%m%d-%H%M%S)"

# 回滚
git checkout backup/before-XXX -- <file>
```

---

## 禁止事项

- ❌ 不重构无关文件
- ❌ 不"顺手优化"
- ❌ 不添加未要求的功能
- ❌ 不修改红线文件
- ❌ 不动认证鉴权
- ❌ 不碰支付财务

## 遇到问题

超出范围时：
```
[红灯] 任务需要改动 [红线/超出预算]
建议：/assess-project 先评估
或：缩小范围，只做核心改动
```
