## 快速开始

### 1. 进入开发

```bash
cd 项目目录
opencode
```

### 2. 新项目初始化

```
/init
```

### 3. 开始需求分析

```
/requirement
描述你的需求
```

### 4. 开始开发

```
/dev 添加用户登录功能
```

## 核心概念

### Change Budget（变更预算）

每次任务必须明确的约束：

```
Change Budget：
- 修改文件数：≤ 3
- 修改行数：≤ 80
- 不引入新依赖
- 不修改任务范围之外的文件
```

### 模块化规范

```
规范文件按需加载，不需要一次性全部读取：
- 做前端 → 加载前端规范
- 做后端 → 加载后端规范
```

### 渐进式文档

```
需求不是一次性写完的，是逐步完善的：
1. 初始需求（用户描述）
2. 结构化文档（AI 生成）
3. 开发计划（任务分解）
4. 变更记录（开发中发现问题）
```

### 项目配置（项目根目录/.opencode/）

```
项目目录/
├── .opencode/
│   ├── commands/          # 项目专属命令
│   ├── skills/            # 项目专属技能
│   └── ...
├── docs/                  # 项目文档
│   ├── requirements.md    # 需求文档
```

```
opencode 启动
    ↓
~/.config/opencode/AGENTS.md  ← 全局（所有项目）
    ↓
项目目录/AGENTS.md             ← 项目级（覆盖全局）
    ↓
opencode.json instructions      ← 额外加载规范文件
    ↓
~/.config/opencode/AGENTS.md  ← Claude Code 兼容
```

### /requirement

**用途**：从用户需求生成结构化需求文档

**用法**：
```bash
opencode
/requirement
我想做一个博客系统，用户可以注册登录，发表文章
```

**输出**：`docs/requirements.md`

**内容**：
- 项目概述
- 模块划分（P0/P1/P2）
- 功能详细说明
- 技术规格（API、数据模型）
- 非功能需求
- 边界定义

---

### /plan-project

**用途**：将需求文档分解为详细开发计划

**用法**：
```bash
opencode
/plan-project
```

**前置条件**：已有 `docs/requirements.md`

**输出**：
- `docs/development-plan.md` - 开发计划
- `docs/progress.json` - 进度状态（自动创建）
- `docs/CHANGELOG.md` - 变更记录（自动创建）

**内容**：
- 开发阶段划分（MVP → 完善 → 优化）
- 具体任务清单
- 优先级排序
- 预计工时

---

### /plan-feature

**用途**：规划单个功能的实现

**用法**：
```bash
opencode
/plan-feature 实现用户登录模块
```

**输出**：实现计划（不实际开发）

**内容**：
- 涉及哪些文件
- 改动范围多大
- 实现步骤
- 简化建议

---

### /dev

**用途**：开始实际开发任务

**用法**：
```bash
opencode
/dev 添加用户注册功能

# 指定更详细的约束
/dev 添加用户登录功能
Change Budget：
- 修改文件数：≤ 2
- 修改行数：≤ 60
```

**输出**：实际代码修改

**约束**：
- 修改文件数 ≤ 3
- 修改行数 ≤ 80
- 不引入新依赖
- 不修改任务范围之外的文件

---

### /assess-project

**用途**：对已有项目进行全面评估，确定AI介入策略和保护机制

**用法**：
```bash
cd 项目目录
opencode
/assess-project
```

**评估流程**：
1. 结构扫描 → 技术栈识别
2. 风险评估 → 核心业务识别
3. 输出项目评估报告

**输出报告内容**：
- 基本信息（项目类型、技术栈、代码规模）
- 风险矩阵（各区域风险等级）
- 推荐介入级别（L1-L5）
- 红线清单（绝对不改的文件/目录）
- 可改区域清单
- 技术债务优先级
- 下一步建议

**适用场景**：
- 新接手一个项目
- 不确定能做什么、不能做什么
- 需要在动手前了解项目风险

### /incremental-dev

**用途**：在已有项目上进行增量开发，遵循保守策略

**用法**：
```bash
opencode
/incremental-dev 修复用户登录页面的样式问题
```

**50行平衡模式规则**：

| 约束 | 限制 |
|------|------|
| 文件数 | ≤ 3 |
| 行数 | ≤ 50 |
| 新依赖 | 不引入 |

**改动前（必须执行）**：
1. 确认改动范围
2. 检查红线文件
3. 自动 git backup

**改动红线（绝对不改）**：
1. 数据库结构/ORM模型
2. 认证鉴权逻辑
3. 支付财务代码
4. 核心业务算法

**改动后**：
- 说明具体改动内容
- 提供验证方法
- 确认功能正常

**适用场景**：
- 修复已有项目的bug
- 在已有项目上添加小功能
- 重构已有项目的局部代码
- 修改已有项目的配置

### /dev

**用途**：开始实际开发任务，自动管理进度

**进度管理**：
- 使用 `docs/progress.json` 记录任务状态
- 支持会话中断/压缩后恢复
- 任务完成后自动更新进度

**用法**：
```bash
opencode
/dev 添加用户注册功能
```

**进度恢复流程**：
```
会话中断/压缩
    ↓
重新执行 /dev
    ↓
读取 docs/progress.json
    ↓
找到 in_progress 任务
    ↓
继续执行
```

### @api-spec-generator

**用途**：生成符合 OpenAPI 3.0 规范的 api-spec.yaml 文件，用于解耦前后端开发

**触发场景**：
- 用户说"帮我生成 API 规范"
- 设计前后端接口
- 需要定义数据模型

**工作流程**：
1. 收集需求（项目类型、接口列表、数据模型）
2. 生成 OpenAPI 3.0 规范的 YAML 文件
3. 校验完整性（路径、方法、schema）
4. 输出 api-spec.yaml

**用途**：
```
api-spec.yaml = 前后端开发的"合同"
    ↓
前端和后端基于同一份规范独立开发
    ↓
最终对接时不再扯皮
```

---

### @keep-focused

**用途**：防止 AI 开发过程中跑偏

**触发**：自动生效，或手动调用

**红色信号（立即拉回）**：

| 信号 | 处理 |
|------|------|
| "顺便..."、"优化了..." | `/undo` + 拒绝 |
| "我重构了..." | `/undo` + 拒绝 |
| 超出 Change Budget | `/undo` + 重新约束 |
| 改任务外文件 | 拒绝 + 重新明确 |

### backend-rules.md

**内容**：

**FastAPI 结构**：
```
backend/
├── main.py          # 入口
├── database.py      # 数据库连接
├── models/         # ORM 模型
├── schemas/         # Pydantic 模型
├── routers/         # 路由
└── crud/            # 数据库操作（可选）
```

**各层职责**：
| 层级 | 职责 | 禁止 |
|------|------|------|
| models/ | 数据库表结构 | 不写业务逻辑 |
| schemas/ | 请求验证、响应 | 不直接操作数据库 |
| routers/ | HTTP 入口 | 不写复杂业务逻辑 |
| crud/ | 数据库操作 | 不写业务规则 |

**什么时候加 crud/**：查询超过 3 行 SQL 或被多处复用

**什么时候加 Service 层**：暂不加，等出现 3+ 处重复业务逻辑

```
│   └── utils/        # 工具函数
```

**平台差异处理**：条件编译（`#ifdef H5` / `#ifdef MP-WEIXIN`）

### typescript-rules.md

**内容**：
- 命名规范（camelCase/PascalCase/UPPER_SNAKE_CASE）
- 类型定义（interface/type/enum）
- 函数规范（箭头函数、泛型）
- Vue 3 + TS（Props/Emits/Computed）
- tsconfig 严格模式

**禁止事项**：
- ❌ 禁止 `any`，用 `unknown` + 类型守卫
- ❌ 禁止 `var`，用 `const` / `let`
- ❌ 禁止隐式 `any`

```
┌─────────────────────────────────────────────────────────────┐
│                    项目开发全流程                              │
└─────────────────────────────────────────────────────────────┘

第 1 步：需求分析
─────────────────────────────────────────────────────────────────────
用户描述需求
    ↓
/requirement
    ↓
提问澄清关键信息
    ↓
生成 docs/requirements.md
    │
    │ 五层结构
    │ 1. 模块划分
    │ 2. 功能项
    │ 3. 具体需求（用户故事、验收标准）
    │ 4. 技术原理（API、数据模型）
    │ 5. 模块关联（依赖关系）
    ↓

第 2 步：计划分解
─────────────────────────────────────────────────────────────
/plan-project 或 @plan-decomposer
    ↓
生成 docs/development-plan.md
    │
    │ 内容
    │ 1. 开发阶段（MVP → 完善 → 优化）
    │ 2. 任务清单
    │ 3. 优先级排序
    │ 4. 预计工时
    ↓

第 3 步：迭代开发
─────────────────────────────────────────────────────────────
循环执行以下步骤：

    ┌─────────────────────────────────────┐
    │ /dev [任务]                         │
    │                                     │
    │ Change Budget：                      │
    │ - 文件数：≤ 3                       │
    │ - 行数：≤ 80                        │
    │ - 不引入新依赖                       │
    └─────────────────────────────────────┘
            ↓
    AI 实施开发
            ↓
    功能验证
            ↓
    ① 正常 → 继续下一个任务
            │
            ↓
    ② 发现问题 → @incremental-doc
    │               ↓
    │           记录到 CHANGELOG.md
    │               ↓
    │           继续开发
    │
    ↓
    ③ AI 跑偏 → @keep-focused
                    ↓
                /undo + 重新约束
                    ↓
                继续开发

第 4 步：会话管理
─────────────────────────────────────────────────────────────
对话超过 10 轮 → /new 新建会话

当前任务完成 → /new 新建会话继续下一个任务

上下文过长 → /compact 压缩上下文

需要回退 → /undo 撤销

第 5 步：提交代码
─────────────────────────────────────────────────────────────
git add .
git commit -m "[类型]: [简短描述]"

类型：feat | fix | docs | style | refactor | perf | test | chore
```

### 三层防护

```
┌─────────────────────────────────────────────────────────┐
│ 第 1 层：AGENTS.md 静态约束                              │
├─────────────────────────────────────────────────────────┤
│ - Change Budget 限制                                    │
│ - 禁止事项清单                                          │
│ - 项目结构定义                                          │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│ 第 2 层：@keep-focused Skill 动态监控                    │
├─────────────────────────────────────────────────────────┤
│ - 检测跑偏信号                                          │
│ - 触发拉回机制                                          │
│ - 验收标准检查                                          │
└─────────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────────┐
│ 第 3 层：开发者主动干预                                  │
├─────────────────────────────────────────────────────────┤
│ - 审视 AI 输出                                          │
│ - 及时 /undo                                            │
│ - 重新明确范围                                          │
└─────────────────────────────────────────────────────────┘
```

### 常见跑偏场景与处理

| 场景 | 处理 |
|------|------|
| AI 说"顺便优化了..." | `/undo` + 拒绝 |
| AI 说"为了扩展性..." | `/undo` + 提醒 YAGNI |
| AI 超出 Change Budget | `/undo` + 重新约束 |
| AI 创建新目录结构 | 确认是否必要 |
| AI 添加新依赖 | 确认是否必要 |
| AI 重构了无关代码 | `/undo` + 拒绝 |

### 开发者口诀

```
每次只做一件事
三文件八十行封顶
AI 想多你要喊停
跑偏就撤重新来
对话太长就新建
复杂功能先计划
简单优先不抽象
```

## 已有项目处理

### 核心挑战

已有项目与新项目有本质区别，处理时需要特别谨慎：

| 维度 | 新项目 | 已有项目 |
|------|--------|----------|
| 起点 | 空白 | 需要理解现有结构 |
| 规范 | 从零贯彻 | 历史代码可能不规范 |
| 改造 | 可完全按规范 | 必须渐进式改造 |
| 目标 | 构建完美 | 保持业务稳定 + 局部优化 |
| AI风险 | 过度设计 | 过度改造/破坏现有逻辑 |

### 介入级别

| 级别 | 场景 | AI权限 | 改动限制 |
|------|------|--------|----------|
| **L1 只读** | 刚接手、不确定风险 | 只读、回答问题 | 不改代码 |
| **L2 诊断** | 定位问题、分析原因 | 分析并输出报告 | 不改代码 |
| **L3 小改** | 改一个函数、一行配置 | 修改代码 | ≤10行，逐次确认 |
| **L4 增量** | 修复bug、加小功能 | 修改代码 | ≤50行，任务完成review |
| **L5 重构** | 较大改动 | 修改代码 | 分阶段授权，明确边界 |

### 改动红线

**绝对不改，除非用户明确授权**：

1. **数据库结构**
   - ORM模型文件（models.py）
   - 数据库迁移文件
   - 数据库连接配置

2. **认证鉴权**
   - 登录/注册逻辑
   - 权限控制代码
   - Token/密钥处理

3. **支付财务**
   - 支付接口
   - 财务计算逻辑
   - 订单处理核心

4. **核心业务**
   - 核心算法
   - 关键业务逻辑

### 标准工作流

```
┌─────────────────────────────────────────────────────────────┐
│                  已有项目处理标准流程                         │
└─────────────────────────────────────────────────────────────┘

第 1 步：项目评估
─────────────────────────────────────────────────────────────
cd 项目目录
opencode
/assess-project
    ↓
分析项目结构、技术栈、风险
    ↓
生成评估报告
    ↓
确定推荐介入级别（L1-L5）
    ↓

第 2 步：增量开发
─────────────────────────────────────────────────────────────
根据评估结果选择介入级别

L1/L2 → 先了解项目，不改代码
L3-L5 → 使用 /incremental-dev 开始改动

/incremental-dev [任务描述]
    ↓
确认改动范围
    ↓
检查红线文件
    ↓
自动 git backup
    ↓
实施改动（≤50行）
    ↓
验证功能
    ↓

第 3 步：遇到问题
─────────────────────────────────────────────────────────────
超出范围 → 先评估再行动
    │
发现新红线 → 停止，询问用户
    │
改动失败 → /undo 回退
    │
遇到风险 → 升级到更高介入级别
```

### 项目评估报告示例

```markdown
# 项目评估报告

## 基本信息
- 项目类型：自研项目
- 技术栈：Python FastAPI + Vue 3
- 代码规模：约 15000 行
- 项目结构：前后端分离

## 风险矩阵
| 区域 | 风险 | 说明 |
|------|------|------|
| models/ | 高 | ORM模型，数据库结构 |
| auth/ | 高 | 认证鉴权逻辑 |
| payment/ | 高 | 支付集成 |
| utils/ | 低 | 工具函数，可改 |
| views/ | 中 | 页面展示，可改需谨慎 |

## 推荐级别：L4 增量

## 红线清单
1. models/user.py - 用户模型
2. models/order.py - 订单模型
3. auth/*.py - 认证模块
4. services/payment.py - 支付服务

## 可改区域
1. utils/ - 工具函数
2. views/ - 页面组件
3. api/ - 接口适配层

## 技术债务优先级
| 优先级 | 问题 | 影响 |
|--------|------|------|
| P1 | utils/ 缺少文档 | 维护困难 |
| P2 | 页面样式不统一 | 需制定规范 |

## 下一步建议
1. 先从 utils/ 入手，添加文档注释
2. 修复视图层的一个具体问题
3. 评估后再决定是否接触核心模块
```

### 适用项目类型

| 项目类型 | 特点 | 建议介入级别 |
|----------|------|--------------|
| 自研项目 | 最熟悉，但风险最高 | L4，从边缘开始 |
| 二次开发 | 有原厂文档参照 | L3，可适度介入 |
| 开源改造 | 有源码可参照 | L4，参照原项目 |

### 禁止事项

- ❌ 不重构无关文件
- ❌ 不"顺手优化"代码
- ❌ 不添加未要求的功能
- ❌ 不修改红线文件
- ❌ 不动认证鉴权
- ❌ 不碰支付财务
- ❌ 不改数据库结构

### 遇到问题时的处理

```
[红灯] 任务需要改动 [红线文件/超出预算]
建议：
1. /assess-project 先评估项目
2. 缩小范围，只做核心改动
3. 升级介入级别，明确授权
```
